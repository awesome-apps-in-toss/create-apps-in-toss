# Barreleye Dashboard PRD v0.2

_One place to manage every mini-app_

---

## 1. 문제 정의

> "앱인토스 콘솔에 등록할 정보가 어디 있더라? PRD는 만들었나? UT는 돌렸나?"

미니앱 수가 늘어날수록 각 앱의 상태를 파악하는 비용이 커진다. 현재 대시보드는 앱 목록을 나열할 뿐, **무엇이 준비됐고 무엇이 빠졌는지 알 수 없다.**

기존 해결책의 한계:

- 파일 시스템 직접 탐색 → 앱마다 구조가 달라 찾기 어려움
- README에 수동 기록 → 업데이트 안 됨, 흩어져 있음
- 기억에 의존 → 앱이 10개 넘으면 한계

---

## 2. 타깃 유저

**Primary**: Barreleye 모노레포를 관리하는 개발자 (1인, 본인)

페르소나:

- 여러 미니앱을 동시에 개발/유지보수
- 앱인토스 콘솔 등록 시 필요한 정보를 모아야 함
- ait-ut, ait-plan, ait-assets 등 스킬을 이미 사용 중
- 브라우저 ↔ 터미널 전환이 잦고 컨텍스트 스위칭 비용이 높음

---

## 3. 핵심 가치 제안

**"각 앱의 준비 상태를 한눈에 보고, 빠진 것은 버튼 하나로 채운다"**

- 3가지 정보 레이어로 무엇이 부족한지 즉시 파악
- 스킬 트리거로 컨텍스트 이탈 없이 에셋/문서 생성
- 생성된 결과물을 대시보드에서 바로 확인 + 콘솔 복붙 준비

---

## 4. 정보 레이어 정의

앱마다 관리해야 할 정보는 세 가지 레이어로 구분된다.

### Layer 1 — 빌드 코드 정보 (진실의 원천: 코드)

코드에 정적으로 적혀 있는 빌드/배포 관련 정보. **반드시 존재해야 하며, 코드가 진실의 원천.**

| 항목         | 파일                                     | 비고               |
| ------------ | ---------------------------------------- | ------------------ |
| appName      | `granite.config.ts`                      |                    |
| displayName  | `granite.config.ts › brand.displayName`  |                    |
| primaryColor | `granite.config.ts › brand.primaryColor` |                    |
| 아이콘 URL   | `granite.config.ts › brand.icon`         |                    |
| `.ait` 파일  | `apps/{앱명}/.ait`                       | 앱인토스 등록 여부 |

대시보드 역할: **읽기 전용 표시** (편집 불가 — 코드를 직접 수정해야 함)

---

### Layer 2 — 콘솔 등록 에셋 (진실의 원천: .meta/console/)

앱인토스 콘솔에 업로드해야 하는 정보. 코드에는 없고 웹 콘솔에서 수동으로 입력/업로드. 귀찮으므로 **스킬로 자동생성 → 대시보드에서 확인 → 콘솔에 복붙/업로드.**

저장 위치: `apps/{앱명}/.meta/console/`

#### 기본 정보

| 항목             | 형식           | 스킬             |
| ---------------- | -------------- | ---------------- |
| 앱 로고          | 이미지 600×600 | `ait-assets` |
| 한국어 앱 이름   | 텍스트         | 수동 입력        |
| 영어 앱 이름     | 텍스트         | 수동 입력        |
| 게임/비게임 분류 | 선택           | 수동 입력        |

#### 카테고리

| 항목              | 형식                               | 비고      |
| ----------------- | ---------------------------------- | --------- |
| 앱인토스 카테고리 | 선택 (e.g. 생활 > 콘텐츠 > 테스트) | 수동 선택 |

#### 노출 정보

| 항목          | 형식             | 스킬               | 비고                                               |
| ------------- | ---------------- | ------------------ | -------------------------------------------------- |
| 부제          | 텍스트           | `ait-plan` 참고 | 사용자가 얻는 가치. 비속어/느낌표 제외             |
| 상세 설명     | 텍스트 (장문)    | `ait-plan` 참고 | 무엇을 보고, 어떤 버튼을 누르고, 무엇을 경험하는지 |
| 검색 키워드   | 쉼표 구분 텍스트 | `ait-plan` 참고 | e.g. 소개팅, 이상형, 테스트                        |
| 가로형 썸네일 | 이미지 1932×828  | `ait-assets`   |                                                    |

#### 미리보기 및 스크린샷

| 항목            | 형식                      | 스킬                    |
| --------------- | ------------------------- | ----------------------- |
| 세로형 스크린샷 | 이미지 636×1048, 최소 3장 | `ait-assets` (추후) |
| 가로형 스크린샷 | 이미지 1504×741, 최소 1장 | `ait-assets` (추후) |

---

### Layer 3 — 개발 헬퍼 (진실의 원천: docs/)

없어도 되지만 있으면 더 좋은 앱을 만들 수 있음. 스킬로 자동생성.

저장 위치:

- PRD: `docs/prd/{앱명}-prd-*.md`
- UT 리포트: `apps/{앱명}/docs/ait-ut-*.md`

| 항목             | 스킬          | 비고                     |
| ---------------- | ------------- | ------------------------ |
| PRD              | `ait-plan` | 제품 요구사항 문서       |
| User Test 리포트 | `ait-ut`      | 사용성 테스트 시뮬레이션 |

## 5. 완성도 계산 기준

레이어별 가중치를 다르게 적용.

```
Layer 1 (빌드 코드)   — granite.config.ts 존재, displayName, primaryColor, icon: 각 10% (40%)
Layer 2 (콘솔 에셋)   — 앱 로고, 부제+설명+키워드, 썸네일: 각 10% (30%)
Layer 3 (개발 헬퍼)   — PRD, UT 리포트: 각 15% (30%)
총합: 100%
```

## 10. 기술 스택

### 프론트엔드

```
React 18 + TypeScript + Vite
React Router v7
```

### 백엔드 (로컬 서버)

```
Express          API 서버 (port 3001)
child_process    Claude Code CLI 실행
SSE              실시간 로그 스트리밍
chokidar         파일 변경 감지
concurrently     pnpm dev 시 프론트 + 서버 동시 실행
```

### 패키지 구성

```
apps/dashboard/
├── src/
│   ├── pages/       Home, AppDetail
│   ├── components/  Sidebar, AppAvatar, CompletionDots, LogStream
│   ├── hooks/       useApps
│   └── types/       AppInfo, GraniteBrand, AppMeta, AppDocs
└── server/
    ├── index.ts
    ├── sse.ts
    ├── watcher.ts
    └── routes/
        ├── apps.ts        # GET /api/apps (Layer 1+2+3 통합)
        ├── meta.ts        # PUT /api/apps/:id/meta (Layer 2 편집)
        └── run-skill.ts   # GET /api/run-skill/stream (스킬 트리거)
```

### 데이터 스키마

**Layer 1** — `granite.config.ts` (읽기 전용)

```ts
{
  appName: string;
  brand: {
    (displayName, primaryColor, icon);
  }
}
```

**Layer 2** — `apps/{앱명}/.meta/app.json`

포맷: **JSON**. 이유: 서버에서 프로그래밍으로 읽고 쓰기 쉬움. TypeScript 타입(`AppConsoleConfig`)은 대시보드 코드에 `interface`로 정의 — 진실의 원천.

에셋 경로(`logoPath`, `thumbnailPath`, `screenshotPaths`)는 **앱 폴더 기준 상대경로**로 저장. 서버가 절대경로로 resolve.

```json
{
  "version": 1,
  "nameKo": "소개팅 발주서",
  "nameEn": "",
  "isGame": false,
  "aitCategory": "생활 > 콘텐츠 > 테스트",
  "subtitle": "",
  "description": "",
  "keywords": ["소개팅", "이상형"],
  "logoPath": ".meta/assets/logo.png",
  "thumbnailPath": ".meta/assets/thumbnail.png",
  "screenshotPaths": [
    ".meta/assets/screenshot-1.png",
    ".meta/assets/screenshot-2.png",
    ".meta/assets/screenshot-3.png"
  ],
  "updatedAt": "2026-04-12T00:00:00Z"
}
```

**Layer 3** — 파일 감지 규칙

```
PRD        docs/prd/{앱명}-prd-*.md
UT 리포트  apps/{앱명}/docs/ait-ut-*.md
```

### 파일 감지 대상

```
apps/*/granite.config.ts
apps/*/.ait
apps/*/.meta/**
apps/*/docs/ait-ut-*.md
docs/prd/**
```

## 11. 리스크 및 가정

| 리스크                      | 가능성 | 대응                                            |
| --------------------------- | ------ | ----------------------------------------------- |
| granite.config.ts 파싱 실패 | 낮음   | 정규식 실패 시 null 처리, 빌드 코드 섹션 미표시 |
| Claude CLI 경로 문제        | 중간   | `which claude`로 경로 검증 후 실행              |
| 스킬 실행 중 에러           | 높음   | stderr 스트리밍 + 실패 상태 표시                |
| 이미지 에셋 경로 싱크       | 중간   | 앱 폴더명 기준으로 절대 경로 통일               |

**가정**

- Claude Code CLI가 로컬에 설치되어 있고 PATH에 등록됨
- 스킬은 앱 폴더 컨텍스트에서 실행 (`cwd: apps/{앱명}`)
- `.meta/` 폴더는 Git에 커밋 (콘솔 정보 버전 관리)
