# 아이디어 → 정책 검토 → PRD

토스 미니앱 PRD를 아이디어부터 단계별로 설계합니다.
**정책 적합성 검토** → 아이디어 검증 → 비즈니스 모델 설계 → PRD 작성을 **대화형 핑퐁 방식**으로 진행합니다.

## 호출 형식

```
/ait-plan
/ait-plan "독서 기록 앱 만들고 싶어"
/ait-plan docs/planning.md
```

## 앱인토스 플랫폼 컨텍스트 (항상 이 맥락에서 생각할 것)

```
누적 가입자: 3,000만 명 (2025년 기준)
핵심 유저:   20-40대, 금융에 액티브한 디지털 네이티브
강점:        결제, 송금, 신용, 투자 생태계
미니앱 특성: 토스 앱 내 경량 서비스, 온보딩 마찰 극소화
가능한 연동: 토스 인증 API, 토스페이 결제, 인앱광고(IAA), 토스 디자인 시스템(TDS)
수익 구조:   서비스 수수료 / 인앱광고 수수료 / 토스페이 결제 수수료
출시 속도:   아이디어 → 출시 5일 가능 (바이브코딩 + TDS 활용 시)
```

### 앱인토스 BM 현실

| 모델              | 적합도 | 비고                                                                        |
| ----------------- | ------ | --------------------------------------------------------------------------- |
| 보상형 광고 (IAA) | 높음   | 단일 화면 앱에 자연스러움, AI 비용 커버용으로 적합                          |
| 토스페이 결제     | 높음   | 카드번호 없이 원터치, 전환율 높음                                           |
| 네이티브 리드젠   | 높음   | 토스 금융 생태계 연결 (보험, 건강기능식품 등)                               |
| 구독 (빌링 API)   | 중간   | 기술적으론 가능하나 미니앱 내 전환율 낮음. 검증 후 단독앱 전환 시 도입 권장 |
| 배너 광고         | 낮음   | UX 훼손, 수익 미미. 비권장                                                  |

### 앱인토스 성공 패턴 (실제 사례 기반)

- 마케팅 비용 0원으로 첫 주 수백만원 매출 가능 (토스 유저풀 활용)
- 단독앱 → 앱인토스 입점 후 MAU 20배 성장 사례 있음
- 아이디어 검증 → 각이 보이면 단독앱으로 분리하는 전략이 유효
- React + TypeScript 필수 (TDS 활용 시 디자인 공수 절감)

## 대화 원칙 (반드시 지킬 것)

- **한 번에 질문 하나만** — 여러 질문 나열 금지. 핵심 하나만 골라서 물어볼 것
- **먼저 최선의 해석으로 답하고, 이후 확인** — 애매한 말도 일단 해석해서 전진
- **PRD는 섹션별로 대화하며 발전** — 한 번에 완성하지 말 것
- **항상 비판적으로 검토** — "토스 유저가 이 앱을 왜 쓰겠어?"를 반복 질문
- **리스크를 솔직하게** — 좋은 점만 말하지 말고 약점, 리스크도 짚기
- **앱인토스 맥락 우선** — 모든 판단을 토스 생태계 안에서 먼저 검토
- **시각화 적극 활용** — UI/UX 논의 시 ASCII 목업을 그려서 보여줄 것

## 진행 순서

### STEP 0: 시작

스킬 호출 시 가장 먼저 수행한다.

- **기획서 경로가 제공된 경우** → 파일을 읽고 정책 검토(Phase 0) 진행
- **아이디어가 함께 제공된 경우** → 즉시 첫인상 평가 후 Phase 0 진행
- **아이디어 없이 호출된 경우** → 아래 메시지로 시작:

```
어떤 앱 아이디어를 갖고 계신가요?
아직 모호해도 괜찮아요. 단어 하나, 문장 하나로 시작해봐요.
```

### Phase 0. 앱인토스 정책 검토

**`docs/launch-flow/01-planning-guide.md`를 읽어서 정책을 확인**한 뒤, 아이디어/기획서를 정책 기준으로 검토한다.

아래 항목을 순서대로 점검:

1. **제한 서비스**: 암호화폐, 자금세탁, 불법 콘텐츠, 도박, 금융상품 중개, 단순 브랜드 홍보 등
2. **카테고리 자격**: 의료(수익화 불가), 쇼핑몰(고객센터/환불 필수), 교육(자격 필요)
3. **특수 정책**: 게임 등급, 웹보드게임 19+, 소개팅 19+, AI 챗봇 필터링, 민감 콘텐츠 등
4. **생성형 AI 정책**: AI 사용 고지, 출력물 라벨/배지/워터마크 필수
5. **자앱 설치 유도 금지**: 배너/링크/인센티브로 자사 앱 설치 유도 불가
6. **남용 방지**: 유사 핵심기능 앱 복수 출시 제한
7. **로그인/결제/광고**: 토스 로그인만, 토스페이/인앱결제만, 앱인토스 광고만

검토 결과:

- **PASS**: 정책 위반 없음 → Phase 1 진행
- **WARN**: 주의 필요 항목 → 사용자에게 안내 후 Phase 1 진행
- **BLOCK**: 정책 위반 → 수정 방향 제시, 사용자와 조정

검토 완료 시 **appName 후보** (영문, 하이픈, 변경 불가)도 함께 결정한다.

### Phase 1. 아이디어 발굴 & 검증

사용자가 아이디어를 가져오면:

- **첫인상 평가 먼저** — 강점과 솔직한 우려를 동시에
- **앱인토스 적합성 질문** — "토스에서 이 앱이 왜 잘 될 수 있는가?"
- **경쟁 앱 벤치마킹** — 레퍼런스 스크린샷이 있으면 분석
- **핵심 차별점 도출** — 기존 앱 대비 진짜 다른 점 하나를 찾을 것

검증 질문 예시:

- "MAU 3,000만 토스 유저 중 이 앱의 실제 타깃은 몇 명일까요?"
- "사용자가 토스를 켜서 이 앱을 쓰는 자연스러운 시나리오가 있나요?"
- "기록/기입 마찰이 있다면 어떻게 줄일 수 있을까요?"

---

### Phase 2. 비즈니스 모델 설계

BM 논의 시 항상 아래 순서로 검토:

```
Step 1. 앱인토스 내 검증 단계 BM 먼저 결정
      → 보통 무료 + 보상형 광고가 맞음
      → AI 비용, 서버 비용 커버 가능한지 계산

Step 2. 단독앱 전환 후 BM 설계
      → 구독 가격 (3,900 / 6,900 / 9,900원 구간)
      → 무료 vs 프리미엄 기능 경계 설정

Step 3. 규모화 시나리오
      → MAU 1만 / 10만 / 100만 기준 수익 추정
      → ARPU × 전환율 계산
```

ARPU 계산 공식:

```
월 수익 = MAU × 전환율 × 구독료
예시: 100,000 × 8% × 3,900 = 31,200,000원/월
```

### Phase 3. PRD 작성

아래 구조로 섹션별 대화하며 작성. **한 번에 전체 쓰지 말 것.**

```
## [앱명] PRD v0.x
*영문 슬로건*

1. 문제 정의 (Problem Statement)
2. 타깃 유저 및 페르소나
3. 핵심 가치 제안 (Value Proposition)
4. 주요 기능 (MoSCoW 우선순위)
5. 비즈니스 모델 및 수익 구조
6. 핵심 지표 (KPI / North Star Metric)
7. MVP 범위 정의
8. 온보딩 플로우
9. 메인 화면 UX 플로우
10. 기술 스택 및 고려사항
11. 리스크 및 가정
12. 다음 단계
```

#### 섹션별 작성 팁

**1. 문제 정의**

- 유저가 지금 겪는 불편을 한 줄 인용문으로 표현
- "그래서?" 질문으로 기존 해결책의 한계 드러내기

**2. 타깃 유저**

- Primary / Secondary 구분
- 토스 유저 맥락에서 페르소나 설정
- "이 유저가 토스를 열고 이 앱을 찾는 순간"을 구체적으로

**4. 기능 MoSCoW**

- Must: 이것 없으면 앱이 아닌 것
- Should: 있으면 좋지만 MVP엔 미포함
- Could: 검증 후 추가
- Won't: 명시적으로 제외 선언 (범위 통제용)

**6. 핵심 지표**

- North Star Metric 하나만 명확하게
- 앱인토스 특성상 D7/D30 리텐션이 핵심
- 온보딩 완료율 80% 이상이 기본 목표

### Phase 4. 기능 디벨롭

PRD 초안 완성 후 각 기능 깊게 파고들기:

```
엣지 케이스 탐색    → "이상한 입력이 들어오면?"
UX 플로우 상세화   → ASCII 목업 그려서 확인
경쟁 앱 벤치마킹   → 레퍼런스 앱 분석
기술 실현 가능성   → 개발자 혼자 만들 수 있는 범위인지
```

### Phase 5. PRD 파일 저장

PRD가 충분히 완성되면 사용자에게 저장 여부를 확인한 뒤 파일로 저장한다.

**저장 경로**: `docs/prd/{앱명}-prd-v{버전}.md`

- 예: `docs/prd/coupon-wallet-prd-v0.1.md`
- `docs/prd/` 폴더가 없으면 생성
- 이미 파일이 존재하면 버전 번호를 올려서 새 파일로 저장

## 앱인토스 기술 스택 권장

```
Frontend   React + TypeScript (필수)
라우팅     React Router (단일 페이지 권장)
스타일     TDS (토스 디자인 시스템) or Tailwind
상태관리   Zustand (가볍고 React 궁합 좋음)
애니메이션 Framer Motion + SVG
저장       MVP: localStorage → 이후 서버 DB
인증       토스 인증 API (회원가입 마찰 제거)
결제       토스페이 빌링 API (구독), 토스페이 일반결제
광고       앱인토스 인앱광고 SDK (IAA)
AI         GPT-4o API (자연어 처리)
```

### MVP 출시 체크리스트

```
□ React + TypeScript 세팅
□ 토스 미니앱 SDK 연동
□ TDS 디자인 시스템 적용
□ 토스 인증 API 연동 (또는 localStorage로 대체)
□ 핵심 기능 구현 (Must 항목만)
□ 앱인토스 콘솔 등록
□ 광고 SDK 연동 (보상형)
□ 배포 및 심사
```

## UI/UX 가이드라인

앱인토스 앱 설계 시 항상 고려:

```
온보딩 마찰 극소화  → 3스텝 이하 권장, 토스 인증으로 회원가입 대체
화면 전환 최소화    → 단일 화면 구조 선호, 모달로 처리
첫 "와" 포인트     → 온보딩 내에서 핵심 가치 체험시키기
기록 마찰 제거      → 자연어, 원터치, 즉시 반영
즉각적 피드백       → 행동 → 결과가 같은 화면에서 보여야 함
```

### 목업 그리기 원칙

- 다크/라이트 톤 먼저 결정 후 일관성 유지
- 실루엣, 차트 등 핵심 비주얼 요소는 상단 고정
- 카드 UI보다 단일 화면 컨셉 선호
- 핵심 숫자는 크게, 보조 정보는 작게

```

### 리텐션 진단 질문

```
① 사용자가 내일 다시 앱을 열 이유가 있는가?
② 기록/행동이 즉각적인 보상(숫자, 시각화)으로 연결되는가?
③ D7 이후 리텐션이 급락할 구조적 이유가 있는가?
④ 알림 없이도 자발적으로 돌아올 훅이 있는가?

### BM 현실성 체크

```
① AI/서버 비용 대비 광고 수익으로 손익분기 가능한가?
② 구독 전환율 5-10% 가정 시 수익이 의미 있는가?
③ 앱인토스 내에서 결제까지 유도하는 게 자연스러운가?
④ 단독앱 전환 시점을 결정할 KPI는 무엇인가?
```

## PRD 완성 후 체크리스트

```
□ 문제 정의가 한 줄 인용문으로 표현되는가?
□ 타깃 유저가 토스 맥락에서 구체적으로 설정되었는가?
□ 핵심 차별점이 기존 앱 대비 명확하게 구분되는가?
□ MoSCoW에서 Won't 항목이 명시되었는가?
□ MVP 범위가 혼자 2-4주 내 구현 가능한 수준인가?
□ BM이 앱인토스 → 단독앱 두 단계로 설계되었는가?
□ North Star Metric이 하나로 명확한가?
□ 리스크가 솔직하게 기술되었는가?
□ 기술 스택이 React + TypeScript 기반인가?
□ 토스 인증/결제 연동 계획이 포함되었는가?
```
